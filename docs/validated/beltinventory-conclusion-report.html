Arcadia Patch Review

Conclusion de test pour MixinBeltInventory

Ce rapport conclut le premier patch retenu pour le mod de performance : un fast-path sur com.simibubi.create.content.kinetics.belt.transport.BeltInventory.tick() quand une belt Create est totalement vide. Les mesures viennent des profils Spark fournis apres test sur serveur avec un seul joueur mais une vraie instance serveur, pas un monde solo local.

Verdict

Decision recommande

Le patch est valide et merite d'etre garde. Le signal local sur la cible belt est tres fort, coherent avec l'intention du patch, et suffisamment net pour etre exploitable meme si le mspt global reste bruite par d'autres facteurs serveur.

Portee

Ce que le patch cible vraiment

  • Belts Create chargees dans le monde
  • Belts structurellement presentes dans une base
  • Mais sans item en transit
  • Sans insertion en attente
  • Sans suppression en attente

Interpretation

Pourquoi ce resultat est credible

  • Le patch est extremement cible.
  • Il court-circuite uniquement un cas totalement vide.
  • Le signal Spark baisse exactement sur la famille de classes attendue.
  • BeltInventory.tick disparait des runs patches, ce qui colle avec un cancel en tete de methode.
Prudence

Ce que ce test ne prouve pas

  • Il ne prouve pas un gain global mspt parfaitement stable.
  • Il ne permet pas de promettre un pourcentage serveur universel.
  • Il n'elimine pas tout le bruit venant d'autres systems charges en meme temps.
Lecture serveur

Ce que cela implique pour un serveur avec plus de joueurs

Le gain global ne sera pas forcement le meme en pourcentage, car il dependra du reste de la charge serveur. En revanche, le gain local observe ici est suffisamment fort pour conclure qu'un serveur avec plus de joueurs, plus de chunks charges et plus de belts presentes beneficiera tres probablement d'un gain concret en temps CPU absolu.

En clair : le patch n'est pas une promesse magique de mspt, mais il retire bien du travail reel sur une cible frequente et pertinente.

Conclusion finale

Position recommandee pour le mod

  • Garder MixinBeltInventory dans le mod.
  • Le considerer comme un patch prudent a fort impact local.
  • Ne pas le vendre comme une optimisation globale universelle, mais comme une micro-optimisation efficace sur les belts Create vides.
  • Utiliser ce patch comme nouvelle base propre avant d'ajouter une seconde cible Create si besoin.

Recommendation de suite logique : conserver ce patch en production de test, puis evaluer ensuite FluidTransportBehaviour avec la meme rigueur.
