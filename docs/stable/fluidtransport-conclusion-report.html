Important Reading

The patch was originally designed as an idle / quasi-idle fast-path, not as a broad active throughput optimization. The live gain is therefore a bonus effect, not the original promise.

The idle comparison is low-sample and therefore weaker. The live comparison is the decisive one here.

The patch should still be described conservatively: it reduces unnecessary pipe-side fluid maintenance when Create enters a truly idle state, and this can also reduce cost in real live networks that repeatedly fall back into idle windows between useful transfers.

Why A Live Gain Is Coherent With The Original Create Code

The local Create 6.0.9 source confirms that the observed live reduction is technically coherent.

  • In FluidTransportBehaviour.tick(), Create always runs the expensive server-side block that scans all connections with getProvidedFluid(), then runs PipeConnection.manageFlows(...), and finally tickFlowProgress(...), once the behaviour has reached UpdatePhase.IDLE.
  • Our patch intentionally injects late, after super.tick(), after the WAIT_FOR_PUMPS -> FLIP_FLOWS -> IDLE transitions, and after manageSource(...). This means we are not skipping initialization or source maintenance.
  • The skip only happens when the behaviour is already in IDLE and all inspected connections report no pressure and no flow.
  • In PumpBlockEntity, the nested PumpFluidTransferBehaviour explicitly extends FluidTransportBehaviour and calls super.tick() before its own pump-specific work. This directly explains why the Spark reductions also show up on pump-related lines.
  • In PipeConnection.manageFlows(...), a pipe may still do meaningful checks even when it ends up doing no useful transfer during that tick. If many ticks land in a state with no pressure and no flow, the fast-path removes that repeated maintenance cost.
  • In FluidNetwork.tick(), the network only becomes relevant after flow and source conditions survive the earlier stages. Reducing idle pipe-side work naturally reduces how often the downstream network path is exercised in the same live setup.
