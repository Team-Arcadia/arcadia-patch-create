Arcadia Perf Patch

Procedure de test pour MixinBeltInventory

Cette procedure sert a valider proprement le premier patch Create retenu dans le mod : un fast-path sur BeltInventory.tick() uniquement quand la belt est totalement vide. L'objectif est de verifier trois choses : stabilite, absence de regression gameplay, et signal perf mesurable.

Objectif

Ce qu'on cherche a prouver

  • Le serveur demarre correctement avec le jar contenant uniquement MixinBeltInventory.
  • Les belts Create actives continuent de fonctionner normalement.
  • Les belts totalement vides consomment moins de temps CPU qu'avant.
  • Le patch n'introduit pas de blocage sur les tunnels, funnels, depots ou processings relies.
Etat attendu

Definition du cas cible

Le patch ne doit s'activer que dans cet etat :

  • items vide
  • toInsert vide
  • toRemove vide

Donc : aucune belt avec items en transit, aucune insertion planifiee, aucune suppression en attente.

Procedure

Plan de test recommande

1. Validation de chargement

Demarrer le serveur avec le jar patché. Confirmer dans le log qu'il n'y a ni crash Mixin ni erreur de target sur Create.

2. Test gameplay simple

Verifier une belt vide, puis une belt avec items, puis une chaine complete avec funnels ou depot. Tout doit fonctionner comme avant.

3. Scenario de bench

Placer ou charger une zone avec beaucoup de belts vides. Garder la zone stable pendant au moins 60 secondes.

4. Capture Spark

Lancer un profil avec patch, puis refaire le meme sans patch si possible dans la meme save et la meme zone.

5. Lecture du resultat

Comparer surtout BeltInventory.tick, puis plus largement les hotspots Create lies aux belts.

Gameplay

Checklist fonctionnelle

  1. Belt totalement vide : pas de comportement anormal, pas de spam log.
  2. Belt avec items entrants : les items avancent normalement.
  3. Funnels et depots : insertion et extraction normales.
  4. Brass tunnels / tri : pas de blocage visible.
  5. Machines de processing sur belt : pas d'item perdu ni fige.
Perf

Checklist Spark

  1. Verifier la presence de com.simibubi.create.content.kinetics.belt.transport.BeltInventory.tick.
  2. Comparer sa part en samples avec et sans patch.
  3. Regarder aussi BeltBlockEntity.tick et SmartBlockEntityTicker.tick.
  4. Eviter de conclure si le monde n'etait pas dans un etat comparable.

Interpretation

Quand considerer le patch comme valide

  • Le serveur demarre sans erreur de mixin.
  • Aucune regression visible sur les belts actives et les blocs relies.
  • Le hotspot BeltInventory.tick baisse ou au minimum ne monte pas de facon suspecte.
  • Le gain observe est coherent avec le scenario vise : beaucoup de belts vides ou quasi inactives.

Si le patch est stable mais que le gain est faible, on peut quand meme le garder comme micro-optimisation conservative. S'il y a un doute gameplay, on le retire.

Suite logique

Ordre recommande apres ce test

  1. Valider ou rejeter MixinBeltInventory.
  2. Si le patch est sain, passer a FluidTransportBehaviour.
  3. Ensuite seulement, et si besoin, investiguer un backoff d'export sur Botany Pots.

Cette procedure est volontairement pragmatique : elle cherche une decision fiable avec un minimum de bruit, pas un benchmark parfait impossible a reproduire.
