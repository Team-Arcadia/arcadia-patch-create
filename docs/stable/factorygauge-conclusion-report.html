Measured Gains

  • static2: about 4.7% lower MSPT than off.
  • static3: about 16.0% lower MSPT than off.
  • adaptive45: about 15.9% lower MSPT than off.
  • adaptive55: about 14.4% lower MSPT than off.
  • static5: still improves MSPT versus off, but the sampled FactoryPanelBehaviour chain becomes worse than baseline, so it is not recommended.

What We Did Not See

  • No clear spike in sampled RedstoneLinkBlock.tick.
  • No clear spike in sampled DisplayLinkBlockEntity work.
  • No sign in this dataset that the patch simply moved cost into a different obvious Create hotspot.

Important Caveat

This benchmark set appears to focus mostly on the local FactoryPanelBehaviour chain. It is still a good validation set, but it is not yet a full proof that every heavy Factory Gauge + Stock Link + Display Link + Redstone Link setup behaves the same way.

Interpretation

The current patch strategy is working as intended. It throttles stable monitoring work instead of touching the action path. In practice, that means the patch reduces repeated storage polling without directly altering the core logistics request flow.

  • static3 is the strongest benchmark mode and should be treated as the reference performance target.
  • adaptive is the best production direction because it keeps most of the gain while staying server-aware.
  • static5 should be avoided; it is the first mode in this set that looks too aggressive.

Recommendation

Keep the patch and continue with adaptive as the recommended live-server mode.

  • Use static3 for repeatable benchmarks and comparisons.
  • Use adaptive for real deployments.
  • Do not recommend static5 in documentation or default settings.
  • Before promoting to stable, one extra live test on a busier chain involving Stock Links or Display Links would still be valuable.
