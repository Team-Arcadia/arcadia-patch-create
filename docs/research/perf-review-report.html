2. Résumé exécutif

Les profils réels des serveurs Arcadia montrent une image cohérente : le serveur dépense surtout son temps dans les ticks monde, les block entities, les entités et les chaînes de transport/logistics. Dans ce cadre, Create ressort plus fortement par ses block entities de transport et ses behaviours, tandis que Mekanism ressort par ses réseaux de transmetteurs. La turbine Mekanism n’apparaît pas comme hotspot majeur sur les serveurs officiels fournis.

3.3 Botany Pots

Les profils montrent bien BotanyPotBlockEntity.tickPot et les chemins de récolte associés (BasicCrop.onHarvest, LootTableDrops.apply). Donc les Botany Pots ne sont pas “invisibles” : ils coûtent quelque chose. En revanche, les données ne démontrent pas encore de manière stricte que le surcoût majeur provient spécifiquement de la combinaison “Botany Pot plein + sortie pleine + câble Mekanism”.

La piste reste néanmoins crédible, parce qu’on voit simultanément :

  • les ticks Botany Pot eux-mêmes,
  • les réseaux Mekanism qui simulent/émettent,
  • et, dans d’autres zones du profil, de fortes chaînes Create logistics.

4. Ce que dit le code source Mekanism

4.1 Réseaux

Dans TransmitterNetworkRegistry.onTick(), Mekanism parcourt tous les réseaux enregistrés et appelle net.onUpdate() à chaque tick serveur. Cela veut dire que dès qu’un grand nombre de réseaux existe, même si chacun n’est “pas énorme”, la somme peut devenir visible.

Dans les trois réseaux principaux :

  • EnergyNetwork.tickEmit() parcourt les acceptors et simule des insertions d’énergie.
  • FluidNetwork.tickEmit() parcourt les acceptors et appelle FluidUtils.canFill().
  • ChemicalNetwork.tickEmit() parcourt les acceptors et appelle ChemicalUtil.canInsert().

Cela confirme que la vraie cible Mekanism est la famille “réseaux/transmitters”, pas UniversalCable en tant que patch exotique et encore moins la turbine comme priorité globale.

4.2 Turbines

Le patch turbine que tu as maintenant est conservateur et techniquement raisonnable. Le problème est seulement son poids relatif : dans les profils réels, la turbine ne pèse pas assez pour être une priorité majeure.

5. Ce que dit le code source Create 6.0.9

5.1 Le ticker “SmartBlockEntity” n’est pas le problème, c’est le conteneur du problème

SmartBlockEntityTicker.tick() se contente d’appeler SmartBlockEntity.tick(). Ensuite, SmartBlockEntity.tick() appelle tous les behaviours du bloc. Donc voir SmartBlockEntityTicker.tick très haut dans Spark signifie simplement “beaucoup de block entities Create font du travail”, pas “ce ticker est mal codé”.

5.2 Fluid transport Create

FluidTransportBehaviour.tick() itère les connexions du bloc et peut passer par :

  • flipFlowsIfPressureReversed()
  • manageSource()
  • manageFlows()
  • tickFlowProgress()

Même quand un pipe ne “produit” rien de visible pour le joueur, il peut continuer à réévaluer sources, pressions, flow state et réseaux internes. C’est une excellente explication de tes observations sur les blocs Create qui tournent “à vide” ou “en surplus”.

5.3 Item logistics Create

Les belts et les blocs autour des belts sont une vraie chaîne de coûts cumulés :

  • BeltInventory.tick() évalue movement, spacing, ending, processing, funnels, tunnels, crushers.
  • FunnelBlockEntity.tick() combine cooldowns, inventaire, checks d’occupation, extraction, insertion belt.
  • ChuteBlockEntity.tick() combine capability lookup, item motion, extraction depuis inventaire et belt.
  • DepotBehaviour.tick() gère items entrants, processing et output belt/funnel.
  • BrassTunnelBlockEntity.tick() fait de la distribution d’outputs et des scans de sorties valides.
  • PackagerBlockEntity.getAvailableItems() et LogisticsManager.getSummaryOfNetwork() relisent des inventaires et résument des réseaux logistics.
Lecture la plus probable
Si tu as l’impression terrain que “les item pipes” sont le vrai problème, les profils et les sources pointent beaucoup plus vers les logistics Create que vers les turbines Mekanism.

7. Décision recommandée

Décision que je recommanderais
Garder le patch turbine uniquement comme micro-patch prudent si tu veux, mais déplacer la priorité de développement vers Create transport/logistics puis vers les réseaux Mekanism. Ne pas ressusciter les anciens patches UniversalCable ou EnergyNetwork basés sur des shadows fragiles.

Ordre de priorité recommandé

  1. Create fluid transport : FluidTransportBehaviour et PipeConnection.
  2. Create item logistics : belts, funnels, chutes, depots, brass tunnels, packagers.
  3. Mekanism transmitters : TransmitterNetworkRegistry, EnergyNetwork, FluidNetwork, ChemicalNetwork.
  4. Validation spécifique éventuelle d’une setup Botany Pots + sortie saturée.
  5. Turbines Mekanism seulement en optimisation secondaire.

8. Conclusion

Si l’objectif est d’obtenir un gain réel sur Arcadia, les données convergent vers une réponse claire : les prochains patchs les plus prometteurs sont du côté de Create transport/logistics et des réseaux Mekanism, pas du côté des turbines.

En d’autres termes : ton intuition “les item pipes foutent le bordel surtout” est bien plus compatible avec les profils réels que l’hypothèse “la turbine est le hotspot principal”.
