Arcadia · Rapport comparatif de stratégies perf

Quoi patcher maintenant, et comment le patcher intelligemment

Ce rapport compare les techniques d’optimisation réellement pertinentes après lecture des profils Spark, des sources locales Mekanism, des sources officielles Create 6.0.9 et des sources officielles Botany Pots 1.21.1. L’objectif est de répondre à une question simple : quels patchs valent vraiment le coup, lesquels sont faisables avec Mixin, et lesquels risquent juste d’être fragiles ou inutiles ?

1. Base de comparaison

La comparaison repose sur quatre familles d’indices.

  • Les profils Spark “officiels” montrent que Create transport/logistics et Mekanism transmitters reviennent de façon répétée.
  • La source Mekanism confirme que les réseaux sont tickés tous les ticks et itèrent les acceptors lors des émissions.
  • La source Create confirme que belts, funnels, chutes, depots, packagers et fluid pipes font du travail récurrent même hors “travail productif” visible.
  • La source Botany Pots confirme qu’un hopper pot mature continue sa boucle de croissance/harvest/export et tente l’insert vers le bas via un cooldown d’export très court.
Idée centrale
Le meilleur patch n’est pas celui qui “supprime du tick” en aveugle, mais celui qui court-circuite proprement des chemins coûteux dans des états déjà stables, saturés ou sans destination utile.

2. Techniques d’optimisation comparées

Technique Principe Valeur Risque gameplay Faisable en Mixin
Early-return sur état idle vérifié Ne rien faire si l’état courant garantit qu’aucun travail utile n’est possible. Très bonne Faible si les conditions sont strictes Oui, excellent candidat
Backoff après échec d’export Si une sortie est pleine et refuse plusieurs fois, espacer les réessais pendant quelques ticks. Très bonne sur systèmes saturés Faible à moyen Oui, très bon candidat
Cache capability / inventory Éviter les lookups répétés tant qu’aucune invalidation n’a eu lieu. Variable Moyen Oui, mais souvent déjà partiellement présent
Time-slicing / round-robin Étaler le travail sur plusieurs ticks au lieu de tout faire chaque tick. Très bonne potentiellement Élevé Oui, mais dangereux pour le gameplay
Augmenter des caches TTL Rendre certaines vues/synthèses moins fraîches mais moins coûteuses. Bonne sur certaines vues Moyen à élevé Oui, mais à manier avec prudence
Patch du ticker générique Réduire le coût d’un ticker central partagé par beaucoup de blocs. Illusoire Élevé Oui, mais mauvaise idée ici

3. Candidats Create

A. Fast-path “belt vide”

Recommandé

Dans BeltInventory.tick(), une grande partie du coût vient de la boucle sur les items transportés et des checks funnel/tunnel/crusher/processing. Si une belt n’a ni item actif, ni insertion en attente, ni suppression en attente, le tick utile est proche de zéro, mais le chemin d’entrée existe quand même.

Pourquoi c’est intelligent : on court-circuite un état structurellement inactif, sans toucher aux belts qui transportent réellement quelque chose.

Risque : faible, si la condition vérifie bien items, toInsert et toRemove. 

@Mixin(value = BeltInventory.class, remap = false)
abstract class MixinBeltInventory {

    @Shadow private List<TransportedItemStack> items;
    @Shadow final List<TransportedItemStack> toInsert;
    @Shadow final List<TransportedItemStack> toRemove;

    @Inject(method = "tick", at = @At("HEAD"), cancellable = true)
    private void mekanismPatch$skipCompletelyEmptyBelt(CallbackInfo ci) {
        if (items.isEmpty() && toInsert.isEmpty() && toRemove.isEmpty()) {
            ci.cancel();
        }
    }
}

B. Fast-path “fluid pipe réellement idle”

Très bon candidat

FluidTransportBehaviour.tick() est un des hotspots Create les plus cohérents dans les profils. Le pattern le plus prometteur est un early-return quand :

  • la phase est déjà IDLE,
  • aucune connexion n’a de pression,
  • aucune connexion n’a de flow actif,
  • aucune animation de progression n’est en cours.

Pourquoi c’est intelligent : ça cible directement les pipes Create “présents mais inertes”, qui correspondent bien à ton observation “blocs Create qui tournent à vide”.

Risque : moyen seulement, car il faut bien respecter les transitions de phase et ne pas casser la reprise quand une pression arrive. 

@Mixin(value = FluidTransportBehaviour.class, remap = false)
abstract class MixinFluidTransportBehaviour {

    @Shadow public Map<Direction, PipeConnection> interfaces;
    @Shadow public FluidTransportBehaviour.UpdatePhase phase;

    @Inject(method = "tick", at = @At("HEAD"), cancellable = true)
    private void mekanismPatch$skipTrulyIdlePipe(CallbackInfo ci) {
        if (phase != FluidTransportBehaviour.UpdatePhase.IDLE || interfaces == null || interfaces.isEmpty()) {
            return;
        }
        for (PipeConnection connection : interfaces.values()) {
            if (connection.hasPressure() || connection.hasFlow()) {
                return;
            }
        }
        ci.cancel();
    }
}

C. Backoff d’extraction pour funnels

Bon candidat, prudence moyenne

FunnelBlockEntity.tick() revient fort dans les profils. Le point sensible n’est pas tant son tick brut que les répétitions de checks quand la destination reste occupée ou pleine. Une stratégie propre est d’ajouter un petit backoff après plusieurs refus consécutifs d’insertion/extraction.

Pourquoi c’est intelligent : ça allège les setups saturées sans ralentir celles qui tournent bien.

Risque : moyen, car toute logique de cooldown touche directement le ressenti d’automation. 

@Mixin(value = FunnelBlockEntity.class, remap = false)
abstract class MixinFunnelBlockEntity {

    @Unique private int mekanismPatch$failedExtractBackoff;

    @Inject(method = "tick", at = @At("HEAD"))
    private void mekanismPatch$decayBackoff(CallbackInfo ci) {
        if (mekanismPatch$failedExtractBackoff > 0) {
            mekanismPatch$failedExtractBackoff--;
        }
    }

    @Inject(method = "activateExtractor", at = @At("HEAD"), cancellable = true)
    private void mekanismPatch$skipRepeatedFailedExtracts(CallbackInfo ci) {
        if (mekanismPatch$failedExtractBackoff > 0) {
            ci.cancel();
        }
    }

    // Après détection d'un refus d'insert/extract côté target, augmenter le backoff.
}

D. Backoff “packager summary”

Possible mais pas premier choix

PackagerBlockEntity.getAvailableItems() et LogisticsManager.getSummaryOfNetwork() apparaissent dans les profils, mais Create utilise déjà VersionedInventoryTrackerBehaviour et TickBasedCache. On peut encore épaissir les caches ou éviter certaines synthèses répétées, mais c’est plus risqué côté cohérence des stocks.

Conclusion : utile seulement après les fast-path belts/fluids, pas avant.

E. Factory Gauge / Display Link logistics fast-path

Bon candidat, à valider par Spark ciblé

Les Factory Gauge ne sont pas un simple affichage “gratuit”. Dans la source Create 6.0.9, FactoryGaugeDisplaySource.provideEntries() parcourt les panels liés puis appelle FactoryPanelBehaviour.getLevelInStorage() et getPromised() via createEntry(), pendant que DisplayLinkBlockEntity.updateGatheredData() pilote la collecte et l’envoi des données. Autrement dit, si une gauge semble “cumuler” du coût, le vrai travail serveur est surtout dans la chaîne DisplayLinkBlockEntity + FactoryPanelBehaviour, pas seulement dans le bloc visuel.

Pourquoi c’est crédible : ça colle à l’observation terrain, au code local Create et aux patterns déjà visibles dans les profils officiels sur la partie logistics.

Cibles Spark à surveiller :

  • com.simibubi.create.content.redstone.displayLink.DisplayLinkBlockEntity.tick
  • com.simibubi.create.content.redstone.displayLink.DisplayLinkBlockEntity.tickSource
  • com.simibubi.create.content.redstone.displayLink.DisplayLinkBlockEntity.updateGatheredData
  • com.simibubi.create.content.redstone.displayLink.source.FactoryGaugeDisplaySource.provideEntries
  • com.simibubi.create.content.redstone.displayLink.source.FactoryGaugeDisplaySource.createEntry
  • com.simibubi.create.content.logistics.factoryBoard.FactoryPanelBehaviour.tickStorageMonitor
  • com.simibubi.create.content.logistics.factoryBoard.FactoryPanelBehaviour.getLevelInStorage
  • com.simibubi.create.content.logistics.factoryBoard.FactoryPanelBehaviour.getPromised

Patch intelligent possible : ne pas toucher au rendu global de la gauge, mais ajouter un fast-path conservateur quand aucun panel lié n’existe, ou quand le support/gauge est dans un état sans changement observable depuis le dernier refresh. Il faut cependant rester prudent car toute logique de cache ou de refresh touche directement la cohérence visuelle et l’état des promesses logistics. 

@Mixin(value = DisplayLinkBlockEntity.class, remap = false)
abstract class MixinDisplayLinkBlockEntity {

    @Inject(method = "updateGatheredData", at = @At("HEAD"), cancellable = true)
    private void mekanismPatch$skipFactoryGaugeWithoutPanels(CallbackInfo ci) {
        if (activeSource instanceof FactoryGaugeDisplaySource
            && factoryPanelSupport.getLinkedPanels().isEmpty()) {
            ci.cancel();
        }
    }
}

Conclusion : à mettre dans la liste “à optimiser”, mais derrière belts et fluid pipes tant qu’un Spark ciblé n’a pas confirmé la part exacte de FactoryPanelBehaviour.

F. Create Nuclear reactor idle / no-output audit

Cible à investiguer sérieusement avant patch

Les profils et les logs locaux confirment la présence de Create Nuclear 1.3.2-beta.3 et font apparaître au moins les classes net.nuclearteam.createnuclear.content.multiblock.controller.ReactorControllerBlockEntity et net.nuclearteam.createnuclear.content.multiblock.core.ReactorCoreEntity. Ton observation en jeu suggère qu’un reactor peut conserver un coût notable même dans un état ressenti comme “quasi vide” ou sans débouché utile.

Pourquoi il faut rester prudent : un reactor qui continue à consommer ses rods et à faire tourner sa logique interne n’est pas réellement idle. Sans lecture de la source NeoForge exacte du mod, il serait trop risqué de proposer un fast-path arbitraire sur un multiblock nucléaire.

Cibles Spark à surveiller :

  • net.nuclearteam.createnuclear.content.multiblock.controller.ReactorControllerBlockEntity
  • net.nuclearteam.createnuclear.content.multiblock.core.ReactorCoreEntity
  • leurs méthodes tick, serverTick, tickServer ou équivalent selon la source réelle

Décision raisonnable à ce stade : le mettre dans “à optimiser”, mais uniquement comme sujet d’audit ciblé. Le prochain pas propre est de récupérer et relire la source NeoForge correspondant à la version réellement chargée avant de parler d’un Mixin.

4. Candidats Mekanism

A. Fast-path “aucun acceptor” sur les réseaux

Excellent candidat

C’est la famille de patchs la plus propre côté Mekanism. Le but est de ne pas entrer dans tickEmit() quand le réseau n’a aucun acceptor réel. C’est déjà présent nativement dans EnergyNetwork localement, et tu l’as déjà ajouté par Mixin sur FluidNetwork et ChemicalNetwork.

Pourquoi c’est intelligent : le gain vient directement du pattern visible dans le code : scan et simulation des acceptors. 

@Redirect(
    method = "onUpdate",
    at = @At(
        value = "INVOKE",
        target = "Lmekanism/common/content/network/FluidNetwork;tickEmit(Lnet/neoforged/neoforge/fluids/FluidStack;)I"
    ),
    remap = false
)
private int mekanismPatch$skipEmitWithoutAcceptors(FluidNetwork instance, FluidStack stack) {
    return instance.getAcceptorCount() == 0 ? 0 : tickEmit(stack);
}

B. Throttling global de TransmitterNetworkRegistry.onTick()

Non recommandé

L’idée d’étaler les réseaux sur plusieurs ticks est tentante, mais elle modifie la cadence globale de propagation/transfert et peut casser le comportement attendu des réseaux. C’est un patch “rentable sur papier” mais dangereux en pratique.

Conclusion : ne pas toucher ce point tant qu’un patch leaf-level plus sûr existe.

C. Fast-path “réseau vide et stable”

Possible, mais secondaire

En théorie, on pourrait tenter un skip sur des réseaux vraiment vides, sans buffer, sans update pending et sans acceptor. En pratique, cela duplique facilement des invariants internes de DynamicBufferedNetwork et devient plus fragile qu’un simple fast-path sur tickEmit.

Conclusion : moins bon ratio risque/gain que les patchs déjà ciblés.

5. Candidats Botany Pots

A. Backoff d’export quand la sortie est bouchée

Très bon candidat

La source montre que le hopper pot tente l’export vers le bas via inventoryInsert à chaque expiration de exportCooldown, puis reset le cooldown. Si le dessous reste plein, le même coût se répète. Le patch le plus propre est d’ajouter un backoff quand aucun slot n’a réellement bougé.

Pourquoi c’est intelligent : il cible exactement la setup que tu soupçonnes sans changer la logique de croissance/récolte.

Risque : faible à moyen, parce qu’on ralentit seulement les réessais quand ils échouent tous. 

@Mixin(value = BotanyPotBlockEntity.class, remap = false)
abstract class MixinBotanyPotBlockEntity {

    @Unique private int mekanismPatch$failedExportBackoff;

    @Inject(method = "tickPot", at = @At("HEAD"))
    private static void mekanismPatch$decayBackoff(Level level, BlockPos pos, BlockState state,
            BotanyPotBlockEntity pot, CallbackInfo ci) {
        MixinBotanyPotBlockEntity self = (MixinBotanyPotBlockEntity) (Object) pot;
        if (self.mekanismPatch$failedExportBackoff > 0) {
            self.mekanismPatch$failedExportBackoff--;
        }
    }

    // Dans le bloc d'export, si aucun slot n'est réellement déchargé vers le dessous,
    // augmenter le backoff de quelques ticks.
}

B. Gel de croissance quand le pot est mûr et totalement saturé

Intelligent mais plus risqué

Une variante plus agressive consiste à détecter un hopper pot déjà mûr, avec stockage interne plein et sortie bouchée, puis à ne plus faire la boucle de croissance/récolte tant qu’aucun changement d’inventaire n’a eu lieu.

Valeur : potentiellement forte sur les usines saturées.

Risque : plus élevé, car on touche directement le cycle crop/growth et il faut éviter toute désynchronisation avec un unblock ultérieur.

Conclusion : à tester seulement après un backoff d’export simple.

6. Ce qu’il ne faut pas faire

  • Patch global sur SmartBlockEntityTicker ou SmartBlockEntity.tick() : trop large, trop de faux positifs.
  • Round-robin agressif sur les réseaux Mekanism : trop risqué pour la logique de transfert.
  • Allonger brutalement les caches logistics Create : risque de stocks ou de routes visuellement faux.
  • Réactiver de vieux patches à shadow fragile juste parce qu’un hotspot existe quelque part dans la famille.

8. Conclusion

Conclusion nette
Les patchs les plus intelligents et réellement faisables en Mixin sont ceux qui ajoutent des fast-paths sur des états stables ou des backoffs sur des sorties bouchées. Pour Arcadia, les trois meilleurs candidats sont :
  1. Create BeltInventory fast-path vide
  2. Create FluidTransportBehaviour fast-path idle
  3. Botany Pots backoff d’export quand la sortie reste pleine

Ces trois patchs ont le meilleur ratio gain plausible / sécurité gameplay / faisabilité Mixin. Ils correspondent aussi beaucoup mieux à ce que montrent les profils réels que des patches plus spectaculaires mais dangereux sur la logique interne des réseaux ou des ticks globaux.

Juste derrière, deux sujets doivent désormais rester dans la colonne à optimiser après validation : les Factory Gauge Create, où la vraie cible semble être la chaîne DisplayLinkBlockEntity / FactoryPanelBehaviour, et les reactors Create Nuclear, qui montrent un coût ressenti non négligeable mais demandent encore une lecture Spark ciblée sur leurs block entities avant toute proposition de patch sûre.

Si tu veux maximiser tes chances de succès
Commence par implémenter un patch Create “belt vide”, un patch Create “fluid pipe idle”, puis un patch Botany Pots “failed export backoff”. Ce sont les trois patches les plus rationnels à faire avant toute tentative plus ambitieuse sur les packagers, les funnels ou la planification globale des réseaux.
