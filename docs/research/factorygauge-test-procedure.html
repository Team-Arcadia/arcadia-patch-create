Factory Gauge v2 Validation

Procedure de test complete

Cette procedure doit etre suivie exactement dans l'ordre. Le but est d'obtenir des tests propres, comparables et utilisables. Il ne faut rien improviser pendant les runs.

But exact du test

  • Verifier que le nouveau patch Factory Gauge v2 ne fait plus crash le serveur.
  • Verifier que le nouveau panel admin en inventaire fonctionne correctement.
  • Verifier que les modes off, static et adaptive changent bien le comportement.
  • Mesurer le gain Spark sans casser l'affichage, la redstone ou la logique du gauge.

Regles a respecter

  1. Ne rien construire ni casser pendant un run Spark.
  2. Ne pas changer les reglages au milieu d'un run Spark.
  3. Ne pas ouvrir et fermer plein d'autres interfaces pendant un run Spark.
  4. Toujours attendre 10 secondes apres un changement de mode avant de lancer Spark.
  5. Toujours utiliser la meme commande Spark et la meme duree.
  6. Si le serveur crash, arreter le protocole et garder les logs avant de reprendre.

Si tu changes plusieurs choses a la fois, tu perds la valeur du test.

Ce qu'il faut avoir avant de commencer

  1. Le serveur tourne avec le bon jar.
  2. Tu es dans la zone de test des Factory Gauge.
  3. La zone est deja chargee et stable.
  4. Tu as de quoi recuperer les profils Spark a la fin.
  5. Tu as au moins ces 4 situations en jeu :

Scenario A

Factory Gauge simple, stable, sans Display Link ni Redstone Link.

Scenario B

Factory Gauge relie a un Display Link ou affichage visible.

Scenario C

Factory Gauge relie a une chaine redstone ou un Stock Link.

Scenario D

Factory Gauge dans un etat qui peut reellement changer.

Commandes a utiliser

/arcadiapatchcreate panel
/arcadiapatchcreate status
/arcadiapatchcreate factoryGauge status
/spark profiler start --timeout 60

Les anciennes commandes texte existent toujours, mais pour ce test il faut utiliser le panel inventaire des que possible.

Comment lire le panel admin

Slot Item Action
Belt Patch Bloc vert ou rouge Left click = active ou desactive le patch belt
Fluid Patch Seau ou seau d'eau Left click = active ou desactive le patch fluid
Factory Gauge Mode Barrier / Clock / Comparator Left click = cycle Off -> Static -> Adaptive
Factory Gauge Test Load Repeater Left click = cycle Off -> 35 -> 45 -> 55, Right click = sens inverse
Factory Gauge Status Book Lecture seulement
Admin Panel Help Writable Book Lecture seulement

Etape 1 - Verifier que le panel s'ouvre

  1. Tape exactement : 
    /arcadiapatchcreate panel
  2. Verifie qu'un inventaire s'ouvre.
  3. Verifie que tu vois bien les items du panel.
  4. Ferme le panel.
  5. Tape ensuite : 
    /arcadiapatchcreate status
/arcadiapatchcreate factoryGauge status
  6. Verifie qu'il n'y a aucune erreur serveur.

Si le panel ne s'ouvre pas ou si une erreur apparait, il ne faut pas continuer.

Etape 2 - Verifier que le panel change bien les reglages

  1. Ouvre le panel : 
    /arcadiapatchcreate panel
  2. Clique 1 fois sur Factory Gauge Mode.
  3. Ferme le panel.
  4. Tape : 
    /arcadiapatchcreate factoryGauge status
  5. Verifie que le mode n'est plus off.
  6. Rouvre le panel.
  7. Clique plusieurs fois sur Factory Gauge Mode jusqu'a revenir a off.
  8. Clique sur Factory Gauge Test Load en left click jusqu'a voir 35, puis 45, puis 55.
  9. Fais un right click pour verifier que la valeur revient en arriere.
  10. Ferme le panel.
  11. Tape : 
    /arcadiapatchcreate factoryGauge status
  12. Verifie que le statut correspond exactement a ce que tu viens de faire dans le panel.

Etape 3 - Baseline propre en mode Off

  1. Ouvre le panel.
  2. Regle Factory Gauge Mode sur off.
  3. Regle Factory Gauge Test Load sur Off.
  4. Ferme le panel.
  5. Verifie : 
    /arcadiapatchcreate factoryGauge status
  6. Attends 10 secondes sans toucher a rien.
  7. Lance : 
    /spark profiler start --timeout 60
  8. Pendant 60 secondes :
    • ne casse aucun bloc,
    • ne pose aucun bloc,
    • ne change aucun reglage.
  9. Sauvegarde le profil sous le nom : 
    factorygauge-off

Etape 4 - Test Static 2

  1. Ouvre le panel.
  2. Clique sur Factory Gauge Mode jusqu'a obtenir static.
  3. Si besoin, utilise les commandes texte pour confirmer : 
    /arcadiapatchcreate factoryGauge mode static 2
  4. Ferme le panel.
  5. Verifie : 
    /arcadiapatchcreate factoryGauge status
  6. Attends 10 secondes.
  7. Regarde les scenarios A, B, C et D.
  8. Verifie que rien ne parait faux, gele ou en retard.
  9. Lance : 
    /spark profiler start --timeout 60
  10. Ne touche a rien pendant le run.
  11. Sauvegarde le profil sous : 
    factorygauge-static2

Etape 5 - Test Static 3

  1. Regle exactement : 
    /arcadiapatchcreate factoryGauge mode static 3
  2. Verifie : 
    /arcadiapatchcreate factoryGauge status
  3. Attends 10 secondes.
  4. Regarde encore les scenarios A, B, C et D.
  5. Lance : 
    /spark profiler start --timeout 60
  6. Ne touche a rien pendant le run.
  7. Sauvegarde le profil sous :

Etape 6 - Test Static 5

  1. Regle exactement : 
    /arcadiapatchcreate factoryGauge mode static 5
  2. Verifie : 
    /arcadiapatchcreate factoryGauge status
  3. Attends 10 secondes.
  4. Regarde surtout les scenarios B, C et D.
  5. Si tu vois un retard visuel ou logique, note-le tout de suite.
  6. Lance :

  7. Sauvegarde le profil sous :

Ce qu'il faut regarder pendant chaque run

Element Question a se poser
Factory Gauge simple Est-ce que l'etat affiche reste logique ?
Display Link Est-ce que l'affichage reste lisible et suit encore le changement ?
Stock Link / redstone Est-ce que le signal suit encore le bon etat ?
Changement reel de stock Est-ce que le gauge rattrape normalement sans gel ni rafale brutale ?
Serveur Est-ce qu'il y a une erreur ou un crash dans les logs ?

Ce qu'il faut chercher dans Spark

  • FactoryPanelBehaviour.tickStorageMonitor
  • FactoryPanelBehaviour.getLevelInStorage
  • FactoryPanelBehaviour.getRelevantSummary
  • LogisticsManager.getSummaryOfNetwork
  • RedstoneLinkBlock.tick
  • DisplayLinkBlockEntity.updateGatheredData

Le bon resultat, ce n'est pas juste une baisse sur le gauge. Il faut aussi verifier que le cout ne part pas ailleurs dans la chaine.

Comment conclure

  • Valide : pas de crash, pas de comportement faux, et le cout Spark baisse.
  • A resserrer : gain correct, mais delai visuel ou logique perceptible en modes agressifs.
  • A rejeter : crash, erreur serveur, etat faux, signal faux, ou effet de rattrapage moche.
