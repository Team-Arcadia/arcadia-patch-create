Arcadia Patch Create

Procedure de test pour MixinFluidTransportBehaviour

Cette procedure sert a valider le patch Create applique sur com.simibubi.create.content.fluids.FluidTransportBehaviour.tick(). Le patch est volontairement conservateur : il ne doit annuler le tick que pour des fluid pipes totalement idle, deja en phase IDLE, sans pression et sans flow actif.

Objectif

Ce qu'on veut prouver

  • Le serveur demarre sans erreur Mixin ni regression Create.
  • Les fluid pipes actifs continuent a transporter normalement.
  • Les pipes reellement inertes disparaissent ou baissent nettement dans Spark.
  • La reprise de flux reste immediate quand une pression ou un fluide revient.
Cible

Etat exact vise par le patch

  • phase = IDLE
  • interfaces presentes
  • aucune connexion avec pression
  • aucune connexion avec flow

Ce patch ne cherche pas a accelerer un pipe encore en transition, ni un pipe qui attend encore une propagation utile.

Pre-requis

Environnement de test

Jar build/libs/arcadia-patch-create-1.0.0.jar
Version Create create-1.21.1-6.0.9
Outil de mesure Spark sampler 60 s minimum
Logs utiles latest.log, debug.log, crash reports si besoin
Procedure

Plan de test recommande

  1. Demarrer le serveur avec le jar patché et verifier l'absence d'erreur de chargement Mixin.
  2. Charger une zone avec plusieurs fluid pipes Create, dont une partie reste totalement inactive.
  3. Faire un premier test gameplay sur un pipe actif : transport, reprise, inversion simple si ton setup le permet.
  4. Mettre ensuite le reseau dans un vrai etat idle : pas de pression, pas de flow, pas de nouvelle entrée fluide.
  5. Lancer un Spark 60 s sur cette zone.
  6. Si possible, refaire exactement le meme run sans patch pour comparaison.
Gameplay

Checklist fonctionnelle

  1. Un pipe actif transporte toujours le fluide correctement.
  2. Un pipe qui redevient actif apres idle repart sans blocage.
  3. Les pumps relies n'entrent pas dans un etat incoherent.
  4. Pas de desync visible sur les flows ou les transitions.
  5. Pas de spam log, crash ou comportement bizarre au chunk reload.
Spark

Checklist de lecture

  1. Chercher FluidTransportBehaviour.tick.
  2. Regarder aussi PipeConnection.manageFlows.
  3. Verifier si SmartBlockEntityTicker.tick baisse aussi sur la zone cible.
  4. Comparer surtout un run idle patché contre un run idle sans patch.
Commandes

Sequence Spark conseillee

/spark sampler --timeout 60

Nommage conseille :

fluid-idle-patched.sparkprofile
fluid-idle-unpatched.sparkprofile
fluid-live-patched.sparkprofile
fluid-live-unpatched.sparkprofile
Attention

Points qui peuvent fausser le test

  • Tester un pipe “presque idle” mais encore sous pression.
  • Comparer deux runs sur des setups dont l'etat initial n'est pas le meme.
  • Conclure sur le mspt global sans verifier la cible locale.
  • Confondre un pipe client-only visuellement calme avec un pipe serveur reellement idle.
Critere de validation

Quand garder le patch

Garde le patch si les fluid pipes inertes perdent clairement du poids dans Spark, sans casser le comportement des pipes actifs ni la reprise normale du reseau.

  • Startup propre
  • Aucune regression gameplay evidente
  • Baisse plausible sur FluidTransportBehaviour.tick en idle
  • Pas de doute sur un retard de reprise de flux

Si le patch est stable mais ne montre pas de gain local clair, il faudra le retirer ou le resserrer encore avant de le proposer pour la branche stable.
